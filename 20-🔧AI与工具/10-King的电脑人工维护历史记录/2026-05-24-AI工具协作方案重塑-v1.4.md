## 一、问题背景

### 1.1 发现的痛点

用户（King）在日常使用中积累了大量 AI 工具配置，但存在以下问题：

| 痛点 | 说明 |

| **目录分散** | `.king-ai/`、`.cc-switch/`、`.hermes/`、`.claude/`、`.openclaw/` 等 5+ 个配置目录 |
| **功能重叠** | 用户配置、API Keys、记忆存储、Skills 等功能在多处重复 |
| **历史遗留** | 早期设计的统一配置枢纽（king-ai）已被后续工具取代，但未清理 |
| **协作不明** | Claude Code、Hermes、OpenClaw、cc-switch、Obsidian 之间如何协作不清晰 |

> 我有太多的未记录的早期设计实施成果遗留物，我该怎么挖掘出来，然后重新梳理一套 Claude、Hermes、OpenClaw、cc-switch、Obsidian 的协作运行方案？

## 二、挖掘方法论

### 2.1 三阶段扫描法

```
┌─────────────────────────────────────────────────────────────┐
│  第一阶段：全面扫描                                          │
│  ├── ls -la ~/          → 找到所有隐藏目录                  │
│  ├── find 找数据库/配置   → 统计所有数据文件                 │
│  └── 逐目录查看结构       → 了解目录组织                      │
├─────────────────────────────────────────────────────────────┤
│  第二阶段：依赖分析                                          │
│  ├── ls -la skills/ | grep "^l"  → 找 symlink              │
│  ├── grep 路径引用        → 找配置依赖关系                   │
│  └── 分析数据流向        → 了解谁读写谁                      │
├─────────────────────────────────────────────────────────────┤
│  第三阶段：功能重叠识别                                       │
│  ├── 用户配置  → king-ai vs cc-switch vs hermes             │
│  ├── API Keys → 分散在多处                                   │
│  ├── 记忆存储  → king-ai/memory vs mcp-memory-sqlite        │
│  └── Skills   → 4 个目录各有副本                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 扫描命令汇总

```bash
# 1. 列出所有 AI 相关隐藏目录
ls -la ~/

# 2. 统计 Skills symlink 关系
ls -la ~/.king-ai/skills/ | grep "^l"
ls -la ~/.cc-switch/skills/ | grep "^l"
ls -la ~/.claude/skills/ | grep "^l"

# 3. 查找所有数据库文件
find ~ -name "*.sqlite" -o -name "*.db" 2>/dev/null

# 4. 查找配置文件中的路径引用
grep -r "king-ai\|\.hermes\|\.openclaw" ~/.cc-switch/ 2>/dev/null
grep -r "\.cc-switch" ~/.hermes/ 2>/dev/null

# 5. 统计各目录大小
du -sh ~/.king-ai ~/.cc-switch ~/.hermes ~/.claude ~/.openclaw
```

### 4.2 分工定义（最终版）

| 工具 | 主要职责 | 不再做 | 协作接口 | 状态 |
|------|---------|--------|---------|------|
| **cc-switch** | 模型切换、API 配置、启动管理 | 无 | 其他工具读取其配置 | ✅ 主力 |
| **Claude Code** | 编程开发、代码审查、文件操作 | 重复的记忆存储 | 通过 MCP 与 Hermes 共享记忆 | ✅ 主力 |
| **Hermes Agent** | 即时通讯（飞书/Telegram）、日程管理 | 重复的用户配置 | 通过 MCP 共享记忆 | ✅ 主力 |
| **OpenClaw** | ⚠️ **保留，作为 Hermes 任务执行助手** | - | 可接入 MCP | ✅ 辅助 |
| **Obsidian Vault** | 知识管理、配置备份、可视化层 | 无 | 双工具都写入 Vault | ✅ 核心 |

### 4.3 配置统一方案

| 配置项 | 存储位置 | 格式 | 读取方 |
|--------|---------|------|--------|
| 用户偏好 | `cc-switch/settings.json` | JSON → kingConfig | Hermes, Claude |
| API Keys | `hermes/.env` | ENV | Hermes |
| API Keys 索引 | `Obsidian: 20-AI与工具/00-AI-Shared/api-keys.md` | Markdown | 参考文档 |
| Skills | `cc-switch/skills/` (唯一源) | 目录 | 所有工具 symlink |
| 记忆 | `mcp-memory-sqlite/` (唯一源) | SQLite | 所有工具 MCP |
| 工作日志 | `cc-switch/logs/` (统一) | 日志文件 | 所有工具 |

> **核心洞察**: Self-Evolving Agent 的核心不是"拥有记忆"，而是"通过经验改变行为"
