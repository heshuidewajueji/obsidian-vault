## 一、工具系统概述

Claude Code 内置了完整的**工具系统**，让 AI 能够执行各种操作：

| 工具类型 | 用途 | 示例 |
|----------|------|------|
| **Bash** | 执行终端命令 | `ls`, `git`, `npm` |
| **Read** | 读取文件 | 查看代码、配置 |
| **Write** | 写入文件 | 创建/修改文件 |
| **Edit** | 编辑文件 | 精准修改内容 |
| **Glob** | 查找文件 | 按模式搜索 |
| **Grep** | 搜索内容 | 在文件中查找关键词 |
| **WebFetch** | 获取网页 | 查资料 |
| **WebSearch** | 网络搜索 | 搜索信息 |

## 二、文件操作工具

### Read - 读取文件

```bash
# 读取整个文件
Read file_path="path/to/file.md"

# 读取指定行数
Read file_path="path/to/file.md" limit=100

# 从指定行开始读取
Read file_path="path/to/file.md" offset=50 limit=50
```

**适用场景**：
- 查看文件内容
- 在修改前了解现有代码
- 读取配置文件

### Write - 写入文件

```bash
Write content="# 文件内容\n\n这是新文件" file_path="path/to/new-file.md"
```

**注意**：
- ⚠️ 会覆盖已存在的文件
- ⚠️ 写入前建议先 Read

### Edit - 编辑文件

```bash
# 替换指定内容
Edit file_path="path/to/file.md" old_string="旧内容" new_string="新内容"

# 替换所有匹配项
Edit file_path="path/to/file.md" old_string="旧内容" new_string="新内容" replace_all=true
```

**适用场景**：
- 修改代码中的一行
- 更新配置文件中的某个值
- 精确修改，不影响其他内容

## 三、搜索工具

### Glob - 查找文件

```bash
# 按模式查找
Glob pattern="**/*.js"

# 指定目录
Glob path="src" pattern="**/*.ts"

# 查找特定文件
Glob pattern="**/README.md"
```

**常用模式**：
| 模式 | 说明 |
|------|------|
| `*.js` | 所有 .js 文件 |
| `**/*.md` | 所有 Markdown 文件 |
| `src/**/*` | src 目录下所有文件 |
| `!node_modules/` | 排除 node_modules |

### Grep - 内容搜索

```bash
# 搜索关键词
Grep path="src" pattern="function"

# 显示行号
Grep -n=true path="src" pattern="TODO"

# 正则搜索
Grep path="src" pattern="log\w+\("

# 只显示文件名
Grep output_mode="files_with_matches" path="." pattern="className"
```

**常用选项**：
| 选项 | 说明 |
|------|------|
| `-n` | 显示行号 |
| `-i` | 不区分大小写 |
| `-C 3` | 显示前后 3 行上下文 |
| `--count` | 只显示匹配数量 |

## 四、Bash 终端命令

### 基本命令

```bash
# 列出目录
Bash command="ls -la"

# 执行多条命令
Bash command="cd src && npm install"

# 后台运行
Bash command="npm run dev" run_in_background=true
```

### Git 命令

```bash
# 查看状态
Bash command="git status"

# 查看差异
Bash command="git diff"

# 提交
Bash command="git commit -m '更新内容'"

# 推送
Bash command="git push"
```

### 管道组合

```bash
# 查找并统计
Bash command="ls -la | wc -l"

# 查找特定文件
Bash command="find . -name '*.md' | head -10"
```

## 六、工具使用原则

### 1. 安全第一

| 原则 | 说明 |
|------|------|
| **先读后写** | 修改前先 Read 了解内容 |
| **备份意识** | 重要操作前确认 |
| **精确编辑** | 用 Edit 不用 Write 覆盖 |
| **权限控制** | 不执行未知命令 |

### 2. 效率优先

| 原则 | 说明 |
|------|------|
| **用对工具** | 搜索用 Grep，不用 find |
| **批量操作** | 一次处理多个文件 |
| **缓存利用** | 避免重复读取 |

### 3. 协作规范

| 规范 | 说明 |
|------|------|
| **描述清晰** | 说明要做什么 |
| **提供上下文** | 用 `@` 引用相关文件 |
| **分步执行** | 复杂任务分步骤 |

## 七、实用工作流

### 工作流 1: 修复 Bug

```
1. Grep 搜索错误关键词
2. Read 定位问题文件
3. Edit 精准修改
4. Bash 运行测试
```

### 工作流 2: 添加新功能

```
1. Glob 了解项目结构
2. Read 相关代码
3. Write 新文件或 Edit 修改
4. Bash 运行构建
```

### 工作流 3: 代码审查

```
1. Glob 查找所有源文件
2. Read 逐个审查
3. 汇总问题和建议
```

## 八、常见问题

| 问题 | 解决 |
|------|------|
| Edit 找不到内容 | 检查空格、缩进是否精确 |
| Write 覆盖了文件 | 改用 Edit 精准修改 |
| Grep 没找到结果 | 检查路径和大小写 |
| Glob 模式不生效 | 确认目录和通配符 |

---

## 💡 今日要点

> 1. **Read → Edit → Write** - 修改文件的标准流程
> 2. **Grep 搜索，Glob 找文件** - 用对工具
> 3. **安全第一** - 先读后写，精确编辑

## 📝 个人笔记区（老师已解答）

1、关于工具使用，我该怎么用自然语言写提示词是合理准确的呢？
   > ✅ 回答：
   >
   > **提示词结构公式**：
   > ```
   > [角色] + [任务] + [上下文/文件] + [要求/约束]
   > ```
   >
   > **示例对比**：
   >
   > | 写法 | 效果 |
   > |------|------|
   > | ❌ "帮我看看" | 太模糊，Claude 不知道看什么 |
   > | ❌ "检查代码" | 不明确要检查什么、怎么检查 |
   > | ✅ "帮我检查 `login.js` 文件，找出潜在的 null 异常和性能问题" | 清晰明确 |
   >
   > **好提示词要素**：
   > 1. **明确任务**：做什么？（修复/检查/创建/重构）
   > 2. **提供文件**：用 `@` 引用相关文件
   > 3. **说明要求**：具体标准（如安全性/性能/代码规范）
   > 4. **指定范围**：哪些文件/模块？（可选）
   >
   > **模板示例**：
   > ```
   > @src/utils.js @src/api.js
   > 帮我审查这两个文件的代码质量，
   > 重点关注：1) 错误处理 2) 安全性 3) 代码可读性
   > 输出格式：问题列表 + 改进建议
   > ```
   >
   > **记住**：Claude Code 不知道你的意图，**描述越清楚，结果越准确**

2、我无法准确的描述出我的意图及输出符合提示词结构公式的提示词咋办？
   > ✅ 回答：
   >
   > **这是正常现象，别担心！** 这个问题很常见，有几个解决思路：
   >
   > **思路 1：迭代式对话**
   > - 先说大概想法："我想优化这个登录功能"
   > - Claude 会提问澄清："您想优化哪方面？性能/安全性/用户体验？"
   > - 一步步明确，Claude 会帮你理清思路
   >
   > **思路 2：让 Claude 提问**
   > ```
   > 我不确定怎么描述，我先说一下背景...
   > [描述你的场景]
   > 请问我应该关注什么？怎么优化？
   > ```
   > - Claude 会主动问关键问题
   > - 通过问答帮你明确需求
   >
   > **思路 3：用目标描述代替过程**
   > - ❌ "帮我改这个函数让它更高效" → 你不确定什么是高效
   > - ✅ "这个登录需要 3 秒，太慢了，请优化到 1 秒以内" → 明确目标
   >
   > **思路 4：从结果倒推**
   > - 先描述你看到的问题/错误
   > - "页面报错：Cannot read property 'name' of null"
   > - Claude 会分析原因和解决方案
   >
   > **实用模板（针对你这种情况）**：
   > ```
   > 我的情况：[描述现状]
   > 遇到的问题：[描述遇到什么麻烦]
   > 想要的结果：[描述你希望达到什么效果]
   > 请帮我分析和给出方案
   > ```
   >
   > **记住**：Claude 是你的**助手**，不是专家。它的工作就是帮你把模糊的想法变成清晰的方案。随着对话深入，你会越来越清楚自己的需求。

3、按照你第二问的解答，我是否都可以采用引导式的方式让claude先帮助我完善我的描述及意图，再进行解决，这样可以节省token吗？有没有哪几种标准的提示词开场白可以让我借鉴？
   > ✅ 回答：
   >
   > **关于 Token 节省**：
   > - **引导式对话会消耗更多 Token**，因为多了"澄清问题"这一轮
   > - 但这比做错了再返工要**节省得多**
   > - 建议：**简单任务直接说**，**复杂任务先澄清**
   >
   > **标准提示词开场白（可直接使用）**：
   >
   > | 开场白 | 适用场景 |
   > |--------|----------|
   > | "我遇到一个问题：[描述现象] 请先帮我分析原因" | 调试/排错 |
   > | "我想做 [功能]，但不确定怎么做，请给我建议" | 需求不明确 |
   > | "我有以下文件 [@文件]，请先理解后再给建议" | 需要先分析 |
   > | "请先问我几个问题，帮我明确需求后再执行" | 完全不确定 |
   > | "我的情况是 [背景]，遇到 [问题]，期望 [结果]，请分析" | 结构化描述 |
   >
   > **推荐的引导式开场白模板**：
   > ```
   > # 模板 A：需求澄清型
   > 我想做 [大概的想法]，但不确定具体方案。
   > 请先问我几个关键问题，帮我明确后再设计解决方案。
   >
   > # 模板 B：分析优先型
   > @相关文件
   > 请先阅读这些文件，理解项目背景后，
   > 问我几个问题来明确需求，再给出方案。
   >
   > # 模板 C：错误诊断型
   > 我遇到了这个错误：[粘贴错误信息]
   > 请先帮我分析可能的原因，
   > 问我几个问题确认情况，再给出修复方案。
   > ```
   >
   > **何时用引导，何时直接说**：
   >
   > | 情况 | 建议 |
   > |------|------|
   > | 简单、明确的任务 | 直接描述 |
   > | 复杂、多步骤任务 | 先 /plan |
   > | 完全不知道怎么描述 | 用引导式模板 |
   > | 需要排错 | 先描述错误现象 |
   > | 时间紧迫 | 直接说，哪怕不完美 |

4、我发现cc-switch有# Claude 提示词管理，这个能起到什么作用？
   > ✅ 回答：
   >
   > **`#` 提示词管理的作用**：
   >
   > 这类提示词通常用于**预设/系统级提示**，让 Claude Code 按照你设定的规则工作。
   >
   > **典型用途**：
   >
   > | 用途 | 示例 |
   > |------|------|
   > | **角色定义** | `# 你是一个代码审查助手` |
   > | **输出格式** | `# 始终用表格输出结果` |
   > | **语言偏好** | `# 回复使用中文` |
   > | **行为约束** | `# 删除文件前必须确认` |
   > | **上下文补充** | `# 项目使用 TypeScript + React` |
   >
   > **在 cc-switch 中可能的配置方式**：
   > ```json
   > {
   >   "prompts": {
   >     "default": "# 使用中文交流\n# 代码使用中文注释",
   >     "code-review": "# 专注于安全性问题\n# 发现问题请列出具体行号"
   >   }
   > }
   > ```
   >
   > **与 CLAUDE.md 的区别**：
   >
   > | 文件 | 作用域 | 内容类型 |
   > |------|--------|----------|
   > | `CLAUDE.md` | 项目级 | 项目规范、技术栈 |
   > | `cc-switch #提示词` | 全局级 | 全局行为规则 |
   >
   > **建议**：
   > - 如果你在使用 cc-switch，把通用规则放在 `#` 提示词中
   > - 项目特定规则放在 `CLAUDE.md` 中
   > - 两者结合使用效果更好
