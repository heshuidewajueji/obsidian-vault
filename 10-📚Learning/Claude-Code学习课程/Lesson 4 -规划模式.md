## 一、什么是规划模式

规划模式（Plan Mode）是 Claude Code 的**核心安全特性**，用于处理复杂任务：

| 特点 | 说明 |
|------|------|
| **分析先行** | 先分析任务，再执行 |
| **分步执行** | 逐步完成任务 |
| **用户确认** | 执行前获得用户批准 |
| **可回退** | 发现问题可调整方案 |

**适用场景**：
- 🔴 复杂/多步骤任务
- 🔴 涉及多个文件的修改
- 🔴 可能破坏性的操作
- 🔴 不确定最佳方案时

---

## 二、如何进入规划模式

### 方法 1: 使用 `/plan` 命令

```
/plan
```

### 方法 2: 描述复杂任务

当任务描述足够复杂时，Claude 会自动建议进入规划模式。

### 方法 3: 用户主动要求

```
帮我规划一下这个项目应该怎么重构
```

## 三、规划模式的工作流程

```
┌─────────────────────────────────────┐
│  1. 进入规划模式                     │
│     /plan 或 描述复杂任务            │
└──────────────┬──────────────────────┘
               ▼
┌─────────────────────────────────────┐
│  2. Claude 分析任务                  │
│     - 理解需求                       │
│     - 分析现状                       │
│     - 设计方案                       │
└──────────────┬──────────────────────┘
               ▼
┌─────────────────────────────────────┐
│  3. 输出执行计划                      │
│     - 分步骤说明                     │
│     - 预估影响                       │
│     - 所需工具/文件                   │
└──────────────┬──────────────────────┘
               ▼
┌─────────────────────────────────────┐
│  4. 用户审批                         │
│     /approve 批准                    │
│     或提供反馈进行调整                │
└──────────────┬──────────────────────┘
               ▼
┌─────────────────────────────────────┐
│  5. 执行计划                         │
│     Claude 按计划分步执行              │
└─────────────────────────────────────┘
```

## 四、规划模式的输出示例

### 示例：重构项目

当输入 `/plan` 后，Claude 会输出类似这样的计划：

```
## 执行计划

### 第一步：分析现状
- 检查项目结构
- 识别依赖关系
- 评估风险

### 第二步：制定方案
- 方案 A：渐进式重构（安全但慢）
- 方案 B：全面重构（快速但风险高）
- 建议采用方案 A

### 第三步：执行步骤
1. 创建备份分支 `git branch backup`
2. 更新依赖配置
3. 重构核心模块
4. 更新测试
5. 验证功能

### 预估时间：30 分钟

是否批准执行？
```

## 五、审批命令

| 命令 | 说明 |
|------|------|
| `/approve` | 批准执行当前计划 |
| `/reject` | 拒绝，修改计划 |
| `继续` | 继续执行下一步 |
| `调整` | 要求修改方案 |

## 六、什么时候用规划模式

### ✅ 应该使用

| 场景 | 原因 |
|------|------|
| 大规模代码重构 | 避免破坏性错误 |
| 数据库迁移 | 数据无价 |
| 删除文件/分支 | 不可逆操作 |
| 新功能开发 | 需要整体规划 |
| 不确定最佳方案 | 获取专业建议 |

### ❌ 不需要使用

| 场景 | 原因 |
|------|------|
| 简单问答 | 太繁琐 |
| 文件读取 | 只读操作 |
| 小修改（1-2行）| 过于简单 |
| 已有明确方案 | 重复规划 |

## 七、EnterPlanMode 工具

在代码中可以使用 `EnterPlanMode` 工具主动进入规划模式：

```javascript
// 当任务复杂时，建议使用
EnterPlanMode({
  description: "重构用户认证模块"
})
```

## 八、实战技巧

### 技巧 1: 提供充分上下文

```
/plan
@README.md @项目架构图.md
帮我分析是否需要重构，以及最佳方案
```

### 技巧 2: 明确约束条件

```
/plan
约束条件：
- 不能中断服务
- 必须保持向后兼容
- 预算 2 天时间

帮我规划重构方案
```

### 技巧 3: 分步审批

对于非常复杂的任务，可以要求分步审批：

```
先做第一步，批准后再继续
```

## 九、常见问题

| 问题 | 解答 |
|------|------|
| 规划太简单/太复杂 | 用反馈调整：「更详细一点」「简化」|
| 不想等规划 | 如果任务简单，直接描述「帮我写一个函数」|
| 规划不合理 | 用 `/reject` 拒绝，说明原因 |
| 想自己执行 | 使用 `/plan --dry-run` 只看计划不执行 |

---

## 💡 今日要点

> 1. **复杂任务 → `/plan`** - 安全第一
> 2. **先分析再执行** - 减少返工
> 3. **审批后再执行** - 控制风险
> 4. **用反馈调整** - 「太简单/太复杂」都可以调整
