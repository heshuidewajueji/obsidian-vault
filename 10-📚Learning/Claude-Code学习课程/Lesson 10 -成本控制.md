## 二、Token 消耗优化

### 2.1 Token 是什么

Token 是文本处理的最小单位：
- 1 个 Token ≈ 4 个字符
- 1 个中文词 ≈ 2-3 个 Token

### 2.2 消耗来源

```
总 Token = 输入 Token + 输出 Token + 上下文 Token
```

| 来源 | 说明 |
|------|------|
| **输入** | 你发送的消息 |
| **输出** | Claude 的回复 |
| **上下文** | 对话历史和文件 |

### 2.3 优化技巧

#### 技巧 1: 清理上下文

```bash
# 使用 /compact 压缩上下文
/compact

# 或开始新会话
/exit
```

#### 技巧 2: 精确引用文件

```
❌ 不好：引用整个项目目录
@src/          # 会读取所有文件

✅ 好：只引用需要的文件
@src/utils.js @src/config.js
```

#### 技巧 3: 分解大任务

```
❌ 不好：一次性处理
帮我重构整个项目

✅ 好：分步骤处理
第一步：分析项目结构
第二步：制定重构计划
第三步：执行重构
```

## 四、节省 Token 策略

### 4.1 输入优化

| 策略 | 节省比例 |
|------|----------|
| 使用 Haiku 处理简单任务 | 50-80% |
| 精确引用文件而不是目录 | 30-50% |
| 删除不必要的历史消息 | 20-40% |

### 4.2 提示词优化

```
❌ 冗长：
请帮我非常非常仔细地检查一下这个代码文件，
看看有没有任何可能存在的问题，谢谢！

✅ 简洁：
检查 src/utils.js 代码问题
```

### 4.3 模型分级使用

```bash
# 简单任务用 Haiku
cc-switch use haiku
claude "解释这个概念"

# 复杂任务切换到 Sonnet/Opus
cc-switch use sonnet
claude "重构这个函数"
```

## 五、实际成本计算

### 5.1 假设场景

| 项目 | 数量 | 模型 | Token/次 |
|------|------|------|-----------|
| 简单问答 | 50次/天 | Haiku | 500 |
| 代码修改 | 20次/天 | Sonnet | 2000 |
| 复杂任务 | 5次/天 | Opus | 5000 |

### 5.2 每日成本估算

```
Haiku: 50 × 500 = 25,000 tokens
Sonnet: 20 × 2000 = 40,000 tokens
Opus: 5 × 5000 = 25,000 tokens
---------------------------
总计: 90,000 tokens/天
```

### 5.3 优化后成本

使用 Haiku 处理更多简单任务后：

```
Haiku: 60 × 500 = 30,000 tokens
Sonnet: 15 × 2000 = 30,000 tokens
Opus: 5 × 5000 = 25,000 tokens
---------------------------
总计: 85,000 tokens/天
节省: ~6%
```
