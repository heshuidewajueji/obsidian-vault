# 12-研发管理方法论

> 来源：《研发管理破局》
> 适合：研发经理、项目经理、SQE

## 研发管理的核心挑战

### 常见问题
- 需求变更频繁
- 技术难度预估不足
- 跨部门协作困难
- 交付压力与质量平衡
- 人员能力参差不齐

---

## 研发管理体系框架

```
战略层：产品规划、技术路线
    ↓
管理层：项目组合、资源调配、流程定义
    ↓
执行层：需求分析、设计开发、测试验证、发布运维
```

---

## 研发流程核心环节

### 1. 需求管理
| 活动 | 输出 |
|-----|------|
| 需求收集 | 用户故事、需求列表 |
| 需求分析 | 需求规格说明书 |
| 需求确认 | 需求基线 |
| 需求变更 | 变更评审记录 |

### 2. 技术方案设计
- 系统架构设计
- 详细设计
- 设计评审（DR）
- DVP（设计验证计划）

### 3. 开发执行
- 任务分解（WBS）
- 代码开发/硬件实现
- 持续集成
- 代码评审（Peer Review）

### 4. 测试验证
```
单元测试 → 集成测试 → 系统测试 → 验收测试
    ↓          ↓          ↓          ↓
  回归测试 ←─────────────────────────
```

### 5. 发布管理
- 发布计划
- 版本管理
- 变更控制
- 应急回滚

## 研发度量体系

### 关键指标（KPI）

| 维度 | 指标 | 行业基准 |
|-----|------|---------|
| 进度 | 准时交付率 | >85% |
| 质量 | 缺陷密度 | <5/KLOC |
| 效率 | 人均产出 | 项目定制 |
| 响应 | 需求变更周期 | <1周 |

### 过程指标
- 需求稳定性指数
- 测试覆盖率
- 缺陷逃逸率
- 返工率

## 研发组织模式

### 1. 职能型
- 适合：大型企业、成熟产品
- 优点：专业深度、资源共享
- 缺点：响应慢、跨部门协调难

### 2. 项目型
- 适合：项目制企业
- 优点：目标明确、快速响应
- 缺点：资源利用率低、缺乏技术积累

### 3. 矩阵型（主流）
- 强矩阵：项目经理主导
- 弱矩阵：职能经理主导
- 平衡矩阵：共同管理

### 4. 敏捷型
- 适合：软件为主、需求变化快
- 机制：Scrum、Sprint、看板

## 技术债务管理

### 什么是技术债务
- 代码质量问题积累
- 文档缺失
- 测试不足
- 架构腐化

### 管理策略
1. **预防**：代码规范、Code Review
2. **识别**：静态分析、技术复盘
3. **偿还**：专项清理、迭代消化
4. **监控**：债务看板、趋势分析

## 研发质量保障

### 全流程质量门

```
需求评审 → 设计评审 → 代码评审 → 测试准入 → 测试报告 → 发布评审
    ↓          ↓          ↓          ↓          ↓          ↓
  质量门1    质量门2    质量门3    质量门4    质量门5    质量门6
```

### 质量活动
- FMEA：设计/过程失效模式分析
- QFD：质量功能展开
- 可靠性设计
- 降额设计
- 冗余设计

## 跨部门协同机制

### 常见问题场景
| 问题 | 根因 | 解决方案 |
|-----|------|---------|
| 需求变更失控 | 变更流程缺失 | CCB变更委员会 |
| 开发测试互推 | 责任不清 | 明确Definition of Done |
| 供应商延迟 | 缺乏预警 | 风险日报机制 |
| 交付质量差 | 验证不充分 | 质量门强制卡控 |

### 有效会议
- 每日站会（15min）
- 周项目例会（1h）
- 月度技术评审会
- 季度复盘会

## Anki记忆卡

> L2级别：Anki记忆卡，巩固核心概念

### 基础概念卡
| 正面 | 背面 |
|-----|------|
| 研发管理的四大核心挑战是什么？ | 需求变更频繁、技术难度预估不足、跨部门协作困难、交付压力与质量平衡 |
| Stage-Gate模型的六个阶段是什么？ | 概念→初步筛选→立项定义→开发→测试验证→发布 |
| Agile的核心价值观来自哪本敏捷宣言？ | 《敏捷宣言》，强调个体与互动高于流程与工具，可工作的软件高于详尽的文档，客户合作高于合同谈判，响应变化高于遵循计划 |
| 技术债务的四个管理步骤是什么？ | 预防（代码规范）→识别（静态分析）→偿还（专项清理）→监控（债务看板） |

### 应用分析卡
| 正面 | 背面 |
|-----|------|
| 矩阵型组织的三种类型及适用场景？ | 强矩阵（项目经理主导，适合项目制）、弱矩阵（职能经理主导，适合职能型）、平衡矩阵（共同管理，主流模式） |
| 设计验证计划DVP的核心目的？ | 在设计阶段就规划好后续验证的方法、资源和时间，确保设计可验证、可追溯 |
| 质量门的"准入"与"准出"有何区别？ | 准入：上一活动是否满足输入标准（如代码评审通过才可集成）；准出：本活动是否满足输出标准可交付下游 |

### 关联辨析卡
| 正面 | 背面 |
|-----|------|
| Agile vs Stage-Gate的核心区别？ | Agile适用于需求变化快的软件项目，采用迭代增量式；Stage-Gate适用于硬件/成熟产品，采用线性顺序式 |
| Scrum中Sprint Review与Sprint Retrospective的区别？ | Sprint Review面向产品演示和反馈（What）；Sprint Retrospective面向过程改进（How） |
| 技术债务与项目风险有何关联？ | 技术债务是已知的积累问题，长期拖慢进度；风险是未发生的潜在问题；债务本身也是风险的重要来源 |

### 记忆技巧
- **技术债务**：想象信用卡欠款——不还就持续产生利息（维护成本），长期不还会导致系统崩溃
- **Stage-Gate**：想象工厂流水线，每个"Gate"都是质量检查站，不过关就返工
- **敏捷Scrum**：想象接力赛，每棒（Sprint）都要交接（Review），同时反思如何跑得更好（Retro）
