# 15-GBT32960车联网平台：电动汽车远程服务与管理国标

## 课程定位

**目标人群**：零跑汽车SQE（转向/制动/底盘电控系统），需要理解车联网平台通信协议

**学习目标**：掌握GB/T 32960.3通信协议结构，理解数据上行/下行流程，建立车联网质量评价能力

**前置要求**：了解新能源汽车基本架构，有TBOX项目经验更佳

**特别说明**：King有TBOX项目经理背景，本课程重点是系统化梳理国标要求

## 一、GB/T 32960标准概述

### 标准发展历程

```
GB/T 32960-2016（第一版）
    ├── 第1部分：总则
    ├── 第2部分：车载终端
    └── 第3部分：通信协议及数据格式（现行）
          ↓
GB/T 32960.3-2025（修订版）
    └── 更新了数据格式和通信要求
```

### 标准定位

**GB/T 32960是国家新能源汽车运行安全管理的核心技术标准**：

| 维度 | 说明 |
|------|------|
| 法规要求 | 新能源汽车必须符合 |
| 管理平台 | 国家/地方监测平台对接 |
| 数据上报 | 实时数据上报要求 |
| 安全监控 | 车辆运行安全监控 |

### 与King工作的关系

```
King负责：转向/制动/底盘电控系统
    │
    ├── 这些系统的故障会影响行车安全
    ├── 故障数据可能通过TBOX上报平台
    └── SQE需要理解：数据如何定义质量问题
```

## 二、GB/T 32960.3通信协议结构

### 协议分层

```
应用层
├── 数据单元格式
├── 命令字定义
└── 数据类型定义
    │
    ▼
传输层（TCP/UDP）
    │
    ▼
网络层（IP）
    │
    ▼
链路层
└── 车载终端 ←→ 平台
```

### 数据流向

```
车载终端（TBOX）
    │
    ├── 采集整车数据
    │   ├── 定位数据
    │   ├── 整车数据
    │   ├── 驱动电池数据
    │   ├── 燃料电池数据
    │   ├── 发动机数据
    │   ├── 电机数据
    │   └── 极值数据
    │
    ├── 数据打包（按GB/T 32960.3格式）
    │
    └── 通过无线网络上传至平台
          │
          ▼
    企业平台（数据处理）
          │
          ▼
    国家平台（数据汇总/监管）
```

## 三、数据单元格式

### 通用数据格式

```
每个数据单元包含：
├── 命令字（1字节）— 标识数据类型
├── 数据长度（2字节）— 数据域字节数
├── 数据域（变长）— 具体数据内容
└── 校验码（1字节）— BCC校验
```

### 主要命令字

| 命令字 | 数据类型 | 上报频率 |
|--------|----------|----------|
| 0x01 | 整车数据 | 1s |
| 0x02 | 驱动电池数据 | 1s |
| 0x03 | 燃料电池数据 | 1s |
| 0x04 | 发动机数据 | 1s |
| 0x05 | 电机数据 | 1s |
| 0x06 | 车辆位置数据 | 1s |
| 0x07 | 极值数据 | 1s |
| 0x08 | 报警数据 | 实时 |
| 0x80 | 平台登录 | 连接时 |
| 0x81 | 平台退出 | 断开时 |
| 0x07 | 数据上报（心跳） | 10s/次 |

## 五、报警数据（故障监控核心）

### 报警级别定义

| 报警级别 | 定义 | 响应要求 |
|----------|------|----------|
| 一级报警 | 严重故障，影响行车安全 | 立即处理 |
| 二级报警 | 重要故障，影响功能 | 24小时内处理 |
| 三级报警 | 一般故障 | 适时处理 |

### 底盘相关报警项

**EPS报警**：

| 报警项 | 条件 | 级别 |
|--------|------|------|
| EPS故障 | EPS控制单元检测到故障 | 一级 |
| 助力异常 | 助力功能异常 | 一级 |
| 传感器故障 | 角度/扭矩传感器异常 | 二级 |

**制动系统报警**：

| 报警项 | 条件 | 级别 |
|--------|------|------|
| 制动系统故障 | 制动系统故障 | 一级 |
| 制动助力不足 | 助力功能异常 | 一级 |
| ABS故障 | ABS功能异常 | 二级 |

**VCU报警**：

| 报警项 | 条件 | 级别 |
|--------|------|------|
| 整车CAN通信故障 | CAN总线通信异常 | 一级 |
| 整车软件异常 | 程序异常 | 一级 |
| 高压继电器故障 | 继电器状态异常 | 二级 |

## 六、数据质量评估（SQE视角）

### 数据质量问题类型

| 问题类型 | 表现 | 根因分析 | SQE行动 |
|----------|------|----------|----------|
| 数据缺失 | 某个数据项无数据 | 传感器/网关故障 | 检查硬件 |
| 数据异常 | 数据超出合理范围 | 传感器漂移/噪声 | 滤波处理 |
| 数据延迟 | 数据上报延迟 | 通信质量差 | 检查网络 |
| 数据错误 | 数据值明显错误 | 软件逻辑错误 | 排查软件 |
| 数据不一致 | 多数据源不一致 | 标定问题 | 交叉验证 |

### SQE在车联网质量中的作用

**整车层面**：

| SQE职责 | 具体工作 |
|---------|---------|
| 传感器质量 | 审核传感器信号精度、可靠性 |
| 网关质量 | 审核网关数据转发完整性 |
| TBOX质量 | 审核数据上报功能和协议符合性 |
| 平台对接 | 验证数据与平台接口的符合性 |

**供应商审核要点**：

| 审核领域 | 审核要点 |
|----------|----------|
| 数据采集 | 传感器精度、数据刷新频率是否符合要求 |
| 数据处理 | 数据滤波、异常值处理是否合理 |
| 数据传输 | 协议格式是否符合GB/T 32960.3 |
| 数据存储 | 本地存储是否可以追溯 |
| 故障诊断 | 故障码定义是否符合标准 |

## 八、实际应用案例

### 案例：TBOX数据异常导致平台报警

**问题描述**：
```
问题：平台持续收到某车型制动系统数据异常报警
影响：国家平台报警率上升，影响企业考核
```

**排查过程**：

| 步骤 | 排查内容 | 发现 |
|------|----------|------|
| 1 | 确认报警车辆 | 所有车辆均报警 |
| 2 | 分析报警数据 | 制动踏板数据周期性跳变 |
| 3 | 检查传感器 | 传感器本身正常 |
| 4 | 检查网关 | 网关数据转发延迟 |
| 5 | 确认根因 | 网关软件对踏板信号滤波参数不当 |

**解决方案**：
- 短期：更新网关软件参数
- 长期：将网关纳入软件变更管理流程

**经验总结**：
- TBOX/网关的软件变更也可能影响数据质量
- SQE需要关注软件变更对数据的影响
